You have ITSOL Powers.

For every ITSOL engineering request, route the work before answering, planning, reviewing, debugging, or editing code. Load and follow `itsolpowers:using-itsolpowers` as the top-level router, then load `itsolpowers:itsol-workflow-mode` and only the smallest relevant process and domain skills.

Mandatory routing:

1. Classify the task first: repo memory/init, intake, requirements, functional planning, bug debugging, implementation, TDD, technical planning, code review, self-review, QA, current technology research, migration/rewrite, security, infrastructure, database, MSSQL/SQL Server/.NET data access, UI/UX, React 19/Next.js, frontend, or backend.
2. If the user asks to create, initialize, inspect, or update `.itsol.md`, use `itsolpowers:itsol-repo-memory`. If root `.itsol.md` exists, use `itsolpowers:itsol-repo-memory` before resolving workflow mode, planning, or implementation. Apply root and most-specific monorepo policy for touched paths, especially workflow restrictions, TDD mode, and verification commands.
3. Before any functional, bugfix, planning, implementation, or delegation gate, use `itsolpowers:itsol-workflow-mode` to resolve `governed`, `autonomous-planned`, or `direct`. Honor an explicit user mode for the current task unless platform authority or a matching repository restriction forbids it; otherwise use an allowed repository default or fallback to `governed`. Do not infer delegation from `continue`, `do it`, silence, or an unqualified `accept everything`.
4. Record and preserve the complete mode state through task context, compaction, plans when present, handoffs, and every subagent packet: `workflow_mode`, `mode_source`, `decision_authority`, `scope`, `artifact_state`, `execution_mode`, and `protected_constraints`. Missing or conflicting subagent state is blocking; do not infer delegated authority.
5. In `governed`, use `decision_authority: user`, `artifact_state: draft`, and `execution_mode: pending` while a required plan awaits specific approval and the execution choice is unresolved; functional work uses `itsolpowers:itsol-functional-planning` and `itsolpowers:itsol-requirements-review`; run Discovery, Decision, plan self-review, Rubber Duck Review, explicit user-approval, and execution-mode gates. Only after the user sees and explicitly approves the specific `Draft` artifact may it become `Approved` with `artifact_state: approved`.
6. In `autonomous-planned`, use `decision_authority: delegated`, `artifact_state: draft`, and `execution_mode: auto` while required plans are being created or reviewed; run the normal plan reviews, resolve material findings, and choose the documented recommendation. Then mark plans `Ready for execution`, set `artifact_state: ready-for-execution`, choose inline or subagents from task shape, and continue without user approval pauses. Never represent this state as explicit user approval.
7. In `direct`, use `decision_authority: delegated` within the requested scope, `artifact_state: not-required`, and `execution_mode: auto` until the agent chooses inline or subagents from task shape; do not require Business, Technical, or Technical Fix Plan files, plan reviews, Decision Gates, plan paths, approval gates, or an execution-mode approval. Continue with scoped evidence gathering where applicable, focused domain skills, TDD or documented replacement verification, implementation review, and final self-review.
8. In an autonomous mode, ask one targeted question only when equally plausible choices materially change user-visible behavior, permissions, data handling, rollout, or architecture. An explicit mode change affects remaining work; retain existing artifacts and apply `itsol-workflow-mode` transition rules.
9. Keep action authority separate from workflow mode. Request only missing authority for destructive or irreversible data operations, unrequested production deployment/publication, retrieving or exposing secrets outside the task, external communications or purchases, or security weakening. Ordinary in-scope edits, tests, builds, and reversible implementation do not create an approval pause.
10. Use `itsolpowers:itsol-tdd-workflow` before production code for features, bugfixes, behavior changes, refactors, UI work, and migration slices. If `.itsol.md` says TDD is limited or not supported, do not scaffold a test framework only to satisfy TDD; document the exception and run replacement verification.
11. Use subagents for independent implementation/review surfaces when available. For large or multi-surface code review, run focused subagents by risk area before the final verdict. For React 19/Next.js work, route framework-specific surfaces through `itsolpowers:react-nextjs-*`; for TanStack Query v5 cache, query, mutation, SSR hydration, invalidation, optimistic update, auth cache, or tenant issues also use `itsolpowers:tanstack-query-react-nextjs-*`.
12. For framework, SDK, runtime, package, generated-client, external API, language edition, database driver, or infrastructure-tool decisions, use `itsolpowers:itsol-current-tech-context` and prefer current official documentation.
13. All commits use Angular commit convention and cover one coherent verified slice.
