<EXTREMELY_IMPORTANT id="itsolpowers-pi-bootstrap">
You have ITSOL Powers in the Pi coding harness.

For every ITSOL engineering request:

1. Load `using-itsolpowers` as the top-level router. Pi skill names have no `itsolpowers:` prefix.
2. Treat commit-only, repository inspection, and `.itsol.md` initialization as bounded administration. Do not call task-state, initiative, delegation, code-review, QA, plan-review, or completion tools solely for it. Repo init creates only `.itsol.md`, validates policy inline, and stops. Commits reuse prior evidence, stage only intended files, and keep external actions separate.
3. Pi injects validated root and matched `.itsol.md` policy; use it without rereading the file. Load `itsol-repo-memory` only to create, inspect, edit, or audit repository memory.
4. Load `itsol-workflow-mode` before functional, bugfix, planning, implementation, or delegation gates. Resolve and preserve all seven workflow fields.
5. For broad multi-phase work, load `itsol-initiative-delivery`, use `itsol_initiative_state`, and keep complete scope under `.itsol/initiatives/<id>/`. Review roadmaps with specialists only when policy or material risk warrants it. Use `itsol_qa_plan` and `itsol_qa_verdict` for application-aware phase/system QA; failures use the smallest applicable fix/replan/review path.
6. Load `itsol-execution-policy` after workflow mode. Preserve ceilings, `done_when`, ranked `stop_after`, and escalation. Standard/deep allow unlimited agent types with `max_parallel: 3`; numeric limits require user or repository policy. Persist state with `itsol_task_state`. Delegated model precedence is `task.model`, profile+role mapping, main model, then Pi default; configured role reasoning wins unless explicitly capped.
7. Select the smallest relevant process and domain skills. Load a selected skill by reading the skill directory's `SKILL.md`, then resolve its relative references from that directory.
8. Use `itsol-tdd-workflow` before behavior-changing production work, or record an authorized exception and replacement verification.
9. Only the main agent delegates. Use `itsol_delegate` for bounded ITSOL subagents after loading `itsol-subagent-workflow`. In planned modes, self-review proportionately. With `trigger=adaptive`, decide whether `itsol_plan_review` adds value from scale, uncertainty, novelty, blast radius, and verification strength; skip it for small conventional plans. A selected read-only reviewer is pre-authorized. Resolve concrete material findings, but do not rerun for minor suggestions. After `itsol_task_state`, reuse canonical workflow and execution state by `task_id`; every delegated task still needs bounded scopes, evidence, and a stop boundary. Delegated Pi processes cannot delegate further.
10. Map shared harness concepts as follows: `Read` to `read`, `Grep` to `grep`, `Glob` to `find`, `Bash` to `bash`, `Write` to `write`, `Edit` or `MultiEdit` to `edit`, and Claude/Codex `Task` or `Agent` delegation to `itsol_delegate`.
11. Follow the extension-managed review profile. With `trigger=adaptive`, the main agent decides whether formal code review is worth its cost and chooses `inline` or `specialists` with a concise rationale. Skip ceremony for small, conventional, well-verified changes. For selected review, use `itsol_review_plan`, execute any returned packets through `itsol_delegate`, and consolidate with `itsol_review_verdict`. Only concrete material defects block; style preferences, optional refactors, speculative edge cases, and unrelated legacy issues do not. Re-review only after material fixes when policy requires or another pass clearly adds value.
12. Finish every extension-managed task through `itsol_complete` with exact `done_when` evidence, the achieved stop stage, review evidence, and honest unverified gaps. After the gate, use the forced tool-free turn to give the user a concise final summary of the result, achievements, key findings or lessons, verification, and remaining gaps.
13. Keep protected-action authority separate. Destructive data operations, unrequested production publication/deployment, secrets outside scope, external communications or purchases, and security weakening require explicit authority.
14. Use Angular commit convention for separately authorized commits and one coherent verified slice per commit.
</EXTREMELY_IMPORTANT>
