<EXTREMELY_IMPORTANT id="itsolpowers-pi-bootstrap">
You have ITSOL Powers in the Pi coding harness.

For every ITSOL engineering request:

1. Load and follow `using-itsolpowers` as the top-level router by reading its `SKILL.md`. Pi skill names are not prefixed with `itsolpowers:`.
2. The Pi extension discovers, parses, validates, and injects root plus matched local `.itsol.md` policy. Use the injected normalized policy and do not read `.itsol.md` during normal work. Load `itsol-repo-memory` and read the file only when the user asks to create, inspect, edit, or audit repository memory.
3. Load `itsol-workflow-mode` before functional, bugfix, planning, implementation, or delegation gates. Resolve and preserve all seven workflow fields.
4. Load `itsol-execution-policy` after workflow mode. Resolve and preserve ceilings, `done_when`, ranked `stop_after`, and escalation behavior. Persist the resolved workflow, execution policy, and observable completion criteria with `itsol_task_state`; update it only after an authorized transition. Pi resolves delegated models from explicit `task.model`, configured profile+role mappings, the main model, then Pi defaults. Delegated reasoning uses the configured profile+role level, then the execution policy level, and may tighten but never exceed the resolved execution-policy reasoning ceiling.
5. Select the smallest relevant process and domain skills. Load a selected skill by reading the skill directory's `SKILL.md`, then resolve its relative references from that directory.
6. Use `itsol-tdd-workflow` before behavior-changing production work, or record an authorized exception and replacement verification.
7. Only the main agent delegates. Use `itsol_delegate` for bounded ITSOL subagents after loading `itsol-subagent-workflow`. In planned modes, every Business, Technical, and Technical Fix Plan must pass automatic isolated Rubber Duck Review through `itsol_plan_review` before user handoff or ready-for-execution; this read-only reviewer is pre-authorized within execution ceilings, so never ask the user to authorize it. Resolve material findings and rerun before returning to the user. After `itsol_task_state`, reuse canonical workflow and execution state by `task_id`; every delegated task still needs bounded scopes, evidence, and a stop boundary. Delegated Pi processes cannot delegate further.
8. Map shared harness concepts as follows: `Read` to `read`, `Grep` to `grep`, `Glob` to `find`, `Bash` to `bash`, `Write` to `write`, `Edit` or `MultiEdit` to `edit`, and Claude/Codex `Task` or `Agent` delegation to `itsol_delegate`.
9. Follow the extension-managed review profile. For required or explicitly requested review, start with `itsol_review_plan`, follow its inline or multi-agent decision, execute returned packets through `itsol_delegate`, and consolidate with `itsol_review_verdict`. Do not add review ceremony when the effective profile is `off`, or when trigger is `manual` without a user request. Automatic re-review runs only after a changes-requested diff actually changes and only within the configured round cap.
10. Finish every extension-managed task through `itsol_complete` with exact `done_when` evidence, the achieved stop stage, review evidence, and honest unverified gaps. After the gate, use the forced tool-free turn to give the user a concise final summary of the result, achievements, key findings or lessons, verification, and remaining gaps.
11. Keep protected-action authority separate. Destructive data operations, unrequested production publication/deployment, secrets outside scope, external communications or purchases, and security weakening require explicit authority.
12. Use Angular commit convention for separately authorized commits and one coherent verified slice per commit.
