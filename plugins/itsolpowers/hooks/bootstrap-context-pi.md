<EXTREMELY_IMPORTANT id="itsolpowers-pi-bootstrap">
You have ITSOL Powers in the Pi coding harness.

For every ITSOL engineering request:

1. Load and follow `using-itsolpowers` as the top-level router by reading its `SKILL.md`. Pi skill names are not prefixed with `itsolpowers:`.
2. First classify commit-only and repository-inspection follow-ups. For an already-produced coherent slice, reuse prior scope and evidence and perform the bounded administrative operation directly. Do not call `itsol_task_state`, initiative, delegation, review, plan-review, or completion tools solely for that operation. Inspect the exact diff, stage only intended files, commit with Angular convention, and keep push/tag/release/deploy separately authorized. Pi automatically disables ITSOL workflow tools for a detected commit-only turn.
3. Pi injects validated root and matched `.itsol.md` policy; use it without rereading the file. Load `itsol-repo-memory` only to create, inspect, edit, or audit repository memory.
4. Load `itsol-workflow-mode` before functional, bugfix, planning, implementation, or delegation gates. Resolve and preserve all seven workflow fields.
5. If the source describes a whole module, application, migration, or multi-phase capability, load `itsol-initiative-delivery`, call `itsol_initiative_state`, and maintain complete-scope durable state under `.itsol/initiatives/<id>/`. Do not silently select one slice and stop.
6. Load `itsol-execution-policy` after workflow mode. Resolve and preserve ceilings, `done_when`, ranked `stop_after`, and escalation behavior. Standard and deep default to unlimited distinct agent identities with `max_parallel: 3`; create a numeric total-agent limit only from explicit user or repository policy. Persist the resolved workflow, execution policy, and observable completion criteria with `itsol_task_state`; update it only after an authorized transition. Delegated model precedence is `task.model`, profile+role mapping, main model, then Pi default. Reasoning may tighten but never exceed policy.
7. Select the smallest relevant process and domain skills. Load a selected skill by reading the skill directory's `SKILL.md`, then resolve its relative references from that directory.
8. Use `itsol-tdd-workflow` before behavior-changing production work, or record an authorized exception and replacement verification.
9. Only the main agent delegates. Use `itsol_delegate` for bounded ITSOL subagents after loading `itsol-subagent-workflow`. In planned modes, every Initiative Roadmap, Business, Technical, and Technical Fix Plan must pass automatic isolated Rubber Duck Review through `itsol_plan_review` before user handoff or ready-for-execution; the read-only reviewer is pre-authorized. Resolve material findings and rerun before user handoff. After `itsol_task_state`, reuse canonical workflow and execution state by `task_id`; every delegated task still needs bounded scopes, evidence, and a stop boundary. Delegated Pi processes cannot delegate further.
10. Map shared harness concepts as follows: `Read` to `read`, `Grep` to `grep`, `Glob` to `find`, `Bash` to `bash`, `Write` to `write`, `Edit` or `MultiEdit` to `edit`, and Claude/Codex `Task` or `Agent` delegation to `itsol_delegate`.
11. Follow the extension-managed review profile. For required or explicitly requested review, start with `itsol_review_plan`, follow its inline or multi-agent decision, execute returned packets through `itsol_delegate`, and consolidate with `itsol_review_verdict`. Do not add review ceremony when the effective profile is `off`, or when trigger is `manual` without a user request. Automatic re-review runs only after a changes-requested diff actually changes and only within the configured round cap.
12. Finish every extension-managed task through `itsol_complete` with exact `done_when` evidence, the achieved stop stage, review evidence, and honest unverified gaps. After the gate, use the forced tool-free turn to give the user a concise final summary of the result, achievements, key findings or lessons, verification, and remaining gaps.
13. Keep protected-action authority separate. Destructive data operations, unrequested production publication/deployment, secrets outside scope, external communications or purchases, and security weakening require explicit authority.
14. Use Angular commit convention for separately authorized commits and one coherent verified slice per commit.
</EXTREMELY_IMPORTANT>
